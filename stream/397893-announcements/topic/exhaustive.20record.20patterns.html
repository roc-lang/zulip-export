Topic: exhaustive record patterns

Richard Feldman (Jun 18 2026 at 11:32):

this is a breaking change which we'd previously talked about trying out, and now it has landed!

the change is pretty simple: previously, if we did this...

{ x, y } = { x: 1, y: 2, z: 3 }

...it worked without any compiler warnings or anything like that, even though not all the fields were used.

Richard Feldman (Jun 18 2026 at 11:32):

in Rust, you'd get a compiler warning unless you did one of these instead:

{ x, y, z: _ } = { x: 1, y: 2, z: 3 }

{ x, y, .. } = { x: 1, y: 2, z: 3 }

Richard Feldman (Jun 18 2026 at 11:35):

the tradeoff between these is:

  • the first design is more concise, because if you only want a subset of the fields, you don't need to write _ or ..
  • the second design is more self-documenting, because you can see at a glance that there are more (potentially interesting) fields available, but which are intentionally unused at the moment
  • the second design also unlocks a new way for the compiler to help you, because it lets you intentionally destructure every single one of the fields in a record (intentionally choose not tot use ..) and now if you ever add another field to that record, the compiler will tell you about all the places where you used this technique and haven't updated to use the new field yet.

Richard Feldman (Jun 18 2026 at 11:35):

I think that last point is particularly interesting. I remember back in my Java days when writing custom "equals" and "hash code" methods, it was a footgun that you could add a new field and forget to incorporate it into your equals/hashing operations, and then they would just silently be broken :sweat_smile:

Richard Feldman (Jun 18 2026 at 11:36):

this is a way to prevent that from happening: inside your custom equals/hashing function, you destructure all the record fields on purpose, so that if you ever add another one, you get a compiler warning saying it's unused, which reminds you to incorporate them into those functions

Richard Feldman (Jun 18 2026 at 11:37):

anyway, even though it's more verbose than the first case, I think the benefits the .. design unlocks are worth trying out - so now we have that design too!
