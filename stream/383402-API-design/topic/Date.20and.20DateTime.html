Topic: Date and DateTime

Isaac Van Doren (Jan 03 2025 at 03:00):

On the topic of time (#API design > Datatype for representing epochs), in addition to Instants/Timestamps, we should have types for dates (yyyy-MM-dd) and date times (yyyy-MM-dd hh:mm:ss).

I propose these be named Date and DateTime.

Other possibilities for naming them are PlainDate and PlainDateTime, or LocalDate and LocalDateTime to communicate that they don't store timezone information, but I don't think these names are helpful enough to justify the longer and less natural names.

Jasper Woudenberg (Jan 03 2025 at 09:07):

I was working on bringing my blog into a static-site platform I'm working on last week. I've blog posts in a directory prefixed with the date, as in yyyy-mm-dd-name-of-the-post.md, and needed to:

  • Parse out the date from the path
  • Sort my posts by date
  • Render the dates using a custom render function

I was loth to pull in a dependency for something so small, so I wrote some hacky Str.splitOn ... code to implement that.

I realized that this particular time-related logic had no dependency on locales or time zone databases. I wondered how many uses are like that. There might be a design where builtins include a Time type that could replace my hacky code, but no functions that require a timezone database, such as conversions between Time and Instant. At that point there might no longer be a need for an external time package, but we would instead have a locale package. Either way, it makes sense to me to test out these ideas in a package first.

I guess the Time type I'm describing here is the same as NaiveDateTime that sky mentioned Elixir has, but I don't think that's necessarily a bad thing. In the API documentation for NaiveDateTime it shows that the type implements all sorts of functions for adding intervals to a NaiveDateTime or getting the duration between two NaiveDateTimes. I'd argue it's those functions that are naive, not the type itself. We could omit those kinds of calculation functions for the Roc Time type, and require application authors to convert Time to Instant (using a timezone) before they can do math with it.

Maybe of interest: I'm porting a small date/time library I wrote for Zig to Roc. The bones are there, Roc's way of dealing with operators I think go well with it! It needs some polish and I have some ideas for supporting iterators in the current API. Current progress and some thoughts on the API design here:
https://git.sr.ht/~jwoudenberg/roc-gregorian

There's a Date type, an opaque wrapper around an I32 representing days since the unix epoch, and a Time type, an opaque wrapper around an I64 representing seconds since the unix epoch. Then there's Time.Zone which is a record with a from_utc and to_utc function, so any platform should be able to provide timezones regardless of how those are represented on the host.

The API is restricted to operations on dates or times that are safe and unambiguous, I think that allows for a pretty tidy API (I know these kinds of libraries can grow massive). Despite that I think it's quite a powerful API. The original version in Zig I wrote for a calendar TUI application that among other things evaluaties recurring icalendar events, so pretty serious date/time-work, and it held up so far.

Jasper Woudenberg (Jul 09 2026 at 19:25):

A lot of date-time libraries have a lot of different things representing some type of timestamp, so then you get into the trouble of coming up with clear names for each of them. In the case of roc-gregorian, because there's just one time representing a time, it made sense to just claim the name Time for it.

I think many date-time libraries are so large because they unwittingly serve a couple of different distinct use cases. Like, I would use Instant if I needed to measure time, maybe in the context of benchmarking, or a tracing library. But why would that need to live in the same library that gets used for building calendar-like things? The fact these usecases use entirely different types shows that really these big time package are secretly multiple libraries bundled together. If Instant forms the basic in some separate time-measuring library, call it stopwatch, that simplifies both stopwatch and the date-time library that now doesn't need to support time-measurement anymore.

For instance, if you don't support time-measuring I don't think you need sub-second precision, which means you can cut a lot of API supporting sub-second time units (milliseconds, nanoseconds), and you can also cut Time needing to support different levels of precision.

With regards to Duration, I thought a lot about this and I think there's more downsides than upsides to having it:

  • For starters I don't want to support "month", "year" durations because they don't have a fixed lengths, so they could not be used unambiguously. For the same reason I also don't want to support "day" in the context of a Time type.
  • I like having a separate Date and Time type, but then that raises a question with regards to how Duration should interact with both of these. It doesn't make sense to add a duration representing fractional days to a Date. Should only Time use Duration? Should Date and Time have their own Duration type?
  • If there's multiple different Duration types I think a lot of API will be consumed converting these different durations into one-another.
  • Given that on the high end I don't want to support durations of a "day" or higher, and on the low and I've cut sub-second durations, that leaves seconds, minutes, and hours. At that point I don't know if it's still worth it. time_a + 5 * 60 might not be super pretty, but I think it's reasonably clear. And now most of the API uses concrete types, which all other things being equal seems nice.
