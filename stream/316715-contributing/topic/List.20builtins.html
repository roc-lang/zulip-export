Claude's summary of a cursory investigation into the CI failure:

Test Result
Main (origin/main, no changes) 15/15 ✓
PR (all changes) 14/15, fails on Variable Redefinition
PR with only Builtin.roc reverted 15/15 ✓

AI Hallucination alert (claude's investigation)

  1. Not a memory-ceiling problem. Bumping playground_exe.initial_memory to 256 MB didn't shift the failure — it's a specific OOB access on cycle 5, not exhaustion.
  2. std.heap.wasm_allocator is NOT a pure bump allocator. It has free lists per power-of-2 size class. free() actually returns slots to the bucket and alloc() reuses them. So this isn't a "playground never frees" leak.
  3. What it IS: Allocations round up to the next power of 2, and freed slots can only be reused inside their size class. When Builtin.bin grows from main → our branch, many allocations shift across size-class boundaries — that pushes the working set into different memory and trips a pre-existing OOB pattern. Reverting Builtin.roc to main passes 15/15, confirming our additions are the trigger but not the bug.
  4. CompilerStageData.deinit looked suspicious at first — line 286-288 will bm.deinit() and free the cached buffer if builtin_module is set. But compileSource never assigns result.builtin_module, so it stays null and that branch is dead in this flow. (Worth noting in the issue as a latent footgun.)
  5. The trap reproduces at "Starting HTML generation" on cycle 5 (5th REPL_STEP). That's exactly where each cycle's local arena and HTML buffers get re-allocated — consistent with a layout-sensitive bug exposed when prior allocations now overlap a different region.

Fixing it properly would mean either (a) replacing wasm_allocator with a GeneralPurposeAllocator (or arena-per-cycle for everything reachable from compiler_data) so size-class fragmentation can't strand pointers across cycles, or (b) tracking down the specific UAF/OOB. Either is a real piece of work — bigger than this PR should swallow.

I am certain that we don't want to replace the wasm_allocator, and of course claude could be way off base, but I thought I'd post this here as background info for anyone looking into the issue.
