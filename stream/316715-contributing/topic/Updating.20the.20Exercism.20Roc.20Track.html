Aurélien Geron (Jun 11 2026 at 11:40):

I've pushed the new branch with 6 ported exercises (accumulate, acronym, affine-cipher, all-your-base, allergies, armstrong-numbers):

https://github.com/exercism/roc/tree/migrate-to-new-roc

If you want to try migrating some exercises:

  • you'll need python with the libraries listed in requirements_generator.txt
  • run bin/fetch_configlet to download the problem specs (used to generate the tests)

Then for each exercise (e.g., foo-bar):

  • edit exercises/practice/foo-bar/.meta/template.j2
  • run bin/generate_tests.py foo-bar
  • edit exercises/practice/foo-bar/.meta/Example.roc
  • run ./bin/verify-exercises foo-bar
  • once the tests pass, edit exercises/practice/foo-bar/FooBar.roc
  • run: find exercises/practice/foo-bar -name "*.roc" -exec roc fmt {} \;
  • triple check :sweat_smile:
  • commit & submit PR for this branch

Since some builtins are still missing, you'll have to implement them locally. Please mark them with a TODO so we can switch to the actual built-in when available.

I find it quite fun and it's a good way to catch compiler bugs.

If you want to check out some of these and tell me what you think, here's how things are organized: let's take rational-numbers, for example. Everything is located under the exercises/practice/rational-numbers directory:

  • the .docs subdirectory contains the instructions for the users. Usually these are generic (i.e., the same for all programming languages), but sometimes there are some Roc-specific instructions in .docs/instructions.append.md so watch out for these.
  • Users have to complete the RationalNumbers.roc type module (which is just a stub).
  • When they submit their code, it will be tested using rational-numbers-test.roc and they will see the error messages, if any. This file is usually generated automatically using the .meta/template.j2 template, based on the standard list of tests for each exercise.
  • One example solution is provided in .meta/Example.roc, but the users never see it: it is only there to run tests when updating the Roc track, so it's not as important as the stub or the tests.

NthPrime :: {}.{
    prime : U64 -> Try(U64, [NoPrime0])
    prime = |number| {
        if number == 0 {
            Err(NoPrime0)
        } else if number == 1 {
            Ok(2)
        } else if number == 2 {
            Ok(3)
        } else {
            find_prime = |primes, index, number2| {
                if primes.len() == number2 {
                    primes
                } else {
                    next_index = index + 2
                    new_primes = {
                        if primes.any(|p| next_index % p == 0) {
                            primes
                        } else {
                            primes.append(next_index)
                        }
                    }
                    find_prime(new_primes, next_index, number2)
                }
            }
            find_prime([2, 3, 5], 5, number)
                .last()
                .map_err(
                    |_| {
                        crash "Unreachable: list cannot be empty"
                    },
                )
        }
    }
}

I had to use number2 because of https://github.com/roc-lang/roc/issues/9690 but since you fixed this issue (thank you!) I removed number2, and it fixes the performance issue.
