Richard Feldman (Jun 16 2026 at 13:24):

souf said:

Richard Feldman said:

you get real stack traces without having to pay an insane up-front performance cost

why would the performance cost be so big?

without going too deep down the coroutines rabbit hole:

  • the way we'd implement this is essentially "pick a chunk of memory and tell the OS to use that as a real, totally normal stack, then after we kick off nonblocking I/O operations, our scheduler tells the OS to go switch over to a different stack that we switched away from earlier, so we're able to keep doing useful work while the I/O operation is waiting on the network or whatever. Then eventually the scheduler switches the stack back to the original one sometime after the I/O operation finished, and that work can continue as normal."
  • if you want to get a stack trace at any point there, it's trivial. You have a real, normal stack! So you just do the normal thing and it works as normal. No extra cost.
  • async in Rust is the fastest implementation I know of (e.g. it's lower overhead than how JS does it) and it's totally different. You don't have real stacks, you have a state machine that determines what to do in between the I/O operations. As soon as you get to an I/O operation, your real call stack stops being a useful history of what calls happened in what order.
  • this approach has some flexibility upsides, but a major downside is that if you ask for a stack trace, what you get is "stack trace going back to the last I/O operation" rather than a full stack trace.
  • it's impossible to recover the stack information after the fact. If you want a full stack trace with all the info you'd normally get, there is only one possible way to do it: every time you do an I/O operation, you defensively go walk the entire stack and write down the entire trace somewhere in memory just in case you want to access it later
  • you have to pay this cost on every I/O operation even if the app never actually needs a stack trace, ever. Because this is so expensive, it's common to not do it and just have bad stack traces in async world. Or you pay the performance cost and can recover the normal-looking traces you get for free with the other approach.

Richard Feldman (Jun 16 2026 at 13:26):

so I think for us, the "stackful" approach is better.

Richard Feldman (Jun 16 2026 at 13:27):

separately, there's the question of how to hook into hosts that are using async, such as JS in the browser where effects like http requests are unavoidably async even if that's not what we're compiling directly to

Richard Feldman (Jun 16 2026 at 13:32):

in general, it's possible for us to do this stackful coroutines approach and then hook into the host's async machinery. On non-wasm targets, this can theoretically be pretty fast, although we'd need a working implementation to properly verify that.

On wasm, it's currently possible but slow because wasm's stack switching proposal (which was in Phase 2 last I checked) hadn't landed yet https://effect-handlers.org/talks/stack-switching-wasmcg-feb2025.pdf
