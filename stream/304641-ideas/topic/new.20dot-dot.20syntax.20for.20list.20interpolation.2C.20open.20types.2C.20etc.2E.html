Topic: new dot-dot syntax for list interpolation, open types, etc.

Fritz Psiorz (Aug 21 2024 at 15:16):

This is an idea for a single new syntax ..x that could be used in various different places of the language to simplify things.

List patterns

This change is the smallest one of them all, and is basically just to allow patterns like [first, ..middle, last] as a short form for [first, .. as middle, last] in when patterns. This is just to stay consistent with list interpolation.

List interpolation

Roc currently has string interpolation, which makes working with strings really nice, but when you're working with lists, you have to piece them together by using List.append, List.concat, etc., which is harder to read and write than interpolation, and which is also possibly less efficient because the final length of the list isn't known when the first part of it is allocated. My proposal is to use the same [first, ..middle, last] syntax also for list interpolation.

With these changes, a simple unoptimized textbook quicksort could look like this:

quickSort = \list ->
    when list is
        [] -> []
        [..rest, pivot] ->
            lessEq = List.keepIf rest (\x -> x <= pivot)
            greater = List.keepIf rest (\x -> x > pivot)

            [..quickSort lessEq, pivot, ..quickSort greater]

The list literal in the end is very readable, and the length of the list is easy for the compiler to know before creating it.

Tag union types

The notation for tag union types in Roc is essentially the same as the notation for lists, so a tag union type is just a list of tags. That's very natural and easy to understand. What isn't necessarily easy to understand is the syntax for open tag unions and for union types.

I would suggest just using the same list interpolation syntax also for types:

WarmColor : [Red, Orange, Yellow]
ColdColor : [Green, Blue, Violet]
Color : [Black, White, ..WarmColor, ..ColdColor]

I consider this to be more readable than the current syntax

Color : [Black, White] WarmColor ColdColor

This would also make open tag unions much easier to understand. Instead of the cryptic syntax [Red, Yellow, Green]a, you could just write [Red, Yellow, Green, ..a] which has an obvious meaning to anybody who knows how to work with lists, and instead of [Red, Yellow, Green]*, you could just write [Red, Yellow, Green, ..] which is much more self-explanatory.

Record types

Obviously, the same syntax could also be used for records, so instead of { first : Str, last : Str }* you'd write { first : Str, last : Str, .. } and instead of { first : Str, last : Str }a you'd write { first : Str, last : Str, ..a }.

Record patterns and literals (optional)

An optional extension to this would be to use the same syntax also in record patterns and literals, which means you could use records very flexibly, including adding and deleting fields.

removeLastName : { first : Str, last : Str, ..a } -> { first : Str, ..a }
removeLastName = \{ first, last : _, ..rest } -> { first, ..rest }

Unpacking/Destructuring Records:

Sam Mohr (Sep 18 2024 at 08:58):

To summarize my understanding of the 4 "needs more discussion" points above:

  • [..quickSort lessEq, pivot, ..quickSort greater]
    • We are starting with [..(quickSort lessEq), pivot, ..(quickSort greater)] for an unambiguous list spread syntax, and will consider allowing removal of the parentheses later
    • We can also support [first, ..rest] syntax instead of [first, .. as rest] in pattern matching
  • { a, b, ..others } pattern
    • For performance reasons, we are avoiding supporting ..others as "a record without the a and b fields". We will instead move forwards with { a, b, .. } as data, which is only a minor change that leverages our existing as data syntax
  • { a, b, ..other } extending record fields
    • For the same reason as above, we shouldn't support combining field sets, but either { a, b, ..other } or { ..other, a, b } as an update to the other record would work.
    • This can wait, since we now have &foo as a secondary shorthand based on the current { other & a, b } syntax. We should make sure we consciously drop that visual consistency when moving to a new record update syntax
  • { a, b, .. } = as the new required syntax for open record destructures
    • This seems to be (mostly) agreed to improve refactoring as we are less likely to forget to destructure newly added fields
