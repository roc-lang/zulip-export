I think we should add the ability to mmap a file in basic cli. It would be mmaped and passed to roc a seamless slice. This means no copy when loading data into roc. Brendan Hansknecht (Jul 18 2024 at 01:04):

So we definitely need at least 2 things:

  1. buffered - Necessary if you want to avoid high memory usage. Also important if file is too large to fit in memory, that can really hurt mmap performance.
  2. mmap - often a great answer for medium to large files. Lazy loading. More overhead due to page faults.
  3. readBytes - simplest for new users. Fastest for small files. Required if the file you are reading might change on disk and you want to load a snapshot of it.

Brendan Hansknecht (Jul 18 2024 at 01:22):

Yeah sharp edges with the mmap are:

  1. if you accidentally cause a copy due to uniqueness mistakes, roc will clone the entire file out of the mmap. Might suddenly have a 14GB file loaded fully into memory.
  2. If the file changes while you are reading it, expect to load chunks from both versions.
  3. If the file is deleted or truncated and you hit a page fault to load more of the file, I'm pretty sure it will just crash.
  4. This one I am not sure of, but theoretically if you have clean pages, the os could just dump them when memory pressure is high. By the time you go to read them again, the data may have changed. So it might be possible to get mutation in a roc list.

Brendan Hansknecht (Jul 18 2024 at 01:23):

Related rust doc:

All file-backed memory map constructors are marked unsafe because of the potential for Undefined Behavior (UB) using the map if the underlying file is subsequently modified, in or out of process. Applications must consider the risk and take appropriate precautions when using file-backed maps. Solutions such as file permissions, locks or process-private (e.g. unlinked) files exist but are platform specific and limited.
