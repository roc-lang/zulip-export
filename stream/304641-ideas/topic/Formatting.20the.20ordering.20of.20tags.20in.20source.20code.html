Topic: Formatting the ordering of tags in source code

Kasper Møller Andersen (Dec 08 2024 at 12:39):

This is a pain point I've personally had many times when writing Elm, but I have no idea if it's just me. So now I'm just putting it out there to see how it goes :big_smile:

The key pain is:
When writing a type consisting of many tags, where the tags have some kind of ordering or grouping in their declaration, I want code that pattern matches on them to have the branches in the same order as the type declares the tags. This makes it much easier to navigate in the pattern match in my experience. It mainly happens with large Msg types, where tags are often grouped informally (i.e. some messages relate to initialization, some relate to a specific piece of UI, and so on). It's also not usually a problem in the early states of writing the code, because it's easy to keep them in sync there. My experience is that they tend to drift as the code gets updated by different people over long periods of time, and I often want to read the pattern match by these groupings. And the grouping gets maintained well in the original type declaration, but various pattern matches do not of course.

This of course only goes when matching on the type doesn't have an inherent ordering you need to use (if you have a default case you can't do full reordering for example).

And naturally there are many kinds of types where I don't care about the ordering, so it's not a universal thing for all types.

So I guess the questions are:

  • Does anyone else feel the same about this?
  • Would this be a decent fit for automatic formatting?
  • Is there some heuristic for deciding when a type should dictate pattern match ordering of tags? Maybe it's just when a type gets enough tags in it that some grouping naturally starts to occur (e.g. more than seven tags enforces ordering)

Have we considered giving people more explicit control over what the compiler does for the cases where it actually matters, and then just leaving normal pattern matching to LLVM to do what it considers best?

So now people can better exercise control when they need it, and if we were to have the formatter reorder tags, it wouldn't touch ordered pattern matches of course. Though I think this makes sense even without the reordering formatter.

Brendan Hansknecht (Dec 08 2024 at 19:43):

Kasper Møller Andersen said:

2) Putting performance back in the picture, I'm a bit worried that we're saying that (a) we expect people to want to order match branches for performance reasons, and (b) the performance characteristic of pattern matching is basically up to LLVM heuristics, which are a black box that will potentially change with every LLVM update.

a) I don't think so. I would expect it to be pretty rare, but it can matter. Most people don't count if branches or avoid short circuit conditionals for performance. I think it is the same for when ... is. Almost no one will care. But it is regularly seen in the wild that when it does matter it can have huge perf implications. In the worst case a switch is a linear list of if conditions. So branch m only ever gets executed after testing branch 0 to n. So having the most run branch first can be very important in that case.

b) yeah, lots of perf is up to the whims of llvm. This is why profiling is so important. There are nobs to play with, but I don't think this is a problem for us to try and solve currently. Rust and c++ both have these problems (bar inline assembly that essentially no one recommends using for perf), but manage great perf anyway. Mojo is looking at trying to solve some of these problems, but it is all way too fine grain and detail heavy to fit a language like roc.
