Richard Feldman (May 13 2023 at 15:36):

part of the reason for this is that I want the rules to be simple:

  • every builtin module is always imported
  • every exposed type in every builtin module is always exposed

Richard Feldman (May 13 2023 at 15:37):

also, as noted in the doc - even if we made an exception for true and false (and again, it is an explicit goal not to have any exceptions) then that still wouldn't help in pattern matching

Richard Feldman (May 13 2023 at 15:37):

because no matter what's exposed, true -> always means "a variable which I have happened to name true right here"

Richard Feldman (May 13 2023 at 15:38):

unless you make true a reserved language keyword which has special meaning in patterns

Richard Feldman (May 13 2023 at 15:40):

so in my mind, the two designs that make sense for Bool are:

  • make true and false be language keywords
  • keep the status quo of Bool.true and Bool.false, which has the upsides of avoiding language keywords (especially language keywords which work differently in pattern matches) and also making it nicer to use tag unions than bools for domain modeling, which is an excellent incentive to create
