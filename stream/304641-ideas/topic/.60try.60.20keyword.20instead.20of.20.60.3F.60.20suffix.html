@Kevin Gillette the reason this is parenthesized is because Richard wants try to be lower precedence than |> so that we can do stuff like

text =
    try File.read! "file.txt" |> Result.mapErr FailedToReadFile
# Think of this like the below
#   try (File.read! "file.txt" |> Result.mapErr FailedToReadFile)

without anything special being required. I agree that this way is more readable to a new user than

text =
    File.read! "file.txt"
    |> try Result.mapErr FailedToReadFile

because now the try isn't matched up with the actual fallible operation, but the error wrapping operation that follows it.

Sam Mohr (Sep 11 2024 at 17:43):

Sky Rose (Sep 18 2024 at 13:08):

I didn't quite think through everything thoroughly, but it seems like all the conditions will make it hard to predict what will happen. Some examples:

try used in either the if or else bodies will render the body as a Result if the body has any definitions, and propagate the error if there are NO definitions

if noDefinitions then
  try result
else
  result = result
  try result

As written, these two branches would behave differently, even though they seem like they should be the same.

If the expression is a |> pipeline:
If try is used in the middle of a pipeline, it is propagated to the result of the pipeline
If try is used at the end of a pipeline, it is propagated to the parent of the pipeline

result
|> try

result
|> try
|> identity

As written, these two pipelines would behave differently, even though adding identity to a pipeline shouldn't affect anything.

I would hope for rules that depend less on everything around the try (like whether there's something after it in the pipeline, or whether the function has a definition elsewhere). Depending on where the try is is fine (whether it's in an if, anywhere in a pipeline, or in a definition).

Richard Feldman (Sep 18 2024 at 15:59):

summary of tradeoffs:

  • we definitely want some form of short-circuiting semantics; I don't think it's a good idea to say "let's just not have try"
  • if we're going to have short-circuiting, then we're already going to have a keyword which affects control flow outside the current expression
  • there's a reasonable question about where that bound ends. return says it can go all the way to the enclosing function, but there's an argument for that bound being too far, and that instead it should only be able to affect control flow in some maximum number of enclosing expressions.
  • we have not found a rule for how to bound it in this way that is simple and understandable.
  • in contrast, the rule for "return to the enclosing function" is extremely simple and easy to understand, and familiar to ~every programmer.
  • separately, we've had requests in the past for early return independent of try considerations, and our attempt to address those use cases with #ideas > "early returns" via formatter has had unintended negative consequences (and might not turn out to be a viable solution)
  • it's fair to point out that return is sometimes genuinely the clearest way to write a piece of code, even though its existence in the language would mean that expressions are less isolated when it comes to control flow
  • separately, beginners coming from imperative languages are familiar with how to use return to exit early from the middle of a nested conditional (for example), and giving them access to that way of writing things reduces Roc's learning curve
  • every mainstream language has return, so introducing it reclaims some strangeness budget

Richard Feldman (Sep 18 2024 at 16:03):

this thread is persuading me that return is a good idea, because:

  • the main argument against return is that we can't tell (without scanning through it) whether certain sufficiently nested expressions might be affecting control flow of an outer expression
  • however, the rules for how to solve this problem seem so complicated that they create a worse problem: I can't even tell what the code is going to do from looking at it without running it
