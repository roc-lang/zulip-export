Topic: Shadowing Sigil

Richard Feldman (Dec 08 2023 at 14:27):

so yesterday I was writing some Roc code and I got an error about shadowing. In this particular case, the error was helpful because I actually hadn't realized I had chosen the same name for two different values in the same scope!

This was an example of Roc's ban on shadowing making my code less error-prone, but we've also seen examples in practice of it making code more error-prone.

One way this can happen is when a value becomes stale and it becomes actively important that it not be used again in the same scope. There's a workaround for this (extract the chunk of code where it's stale into a separate function and call it passing only the non-stale value) but applying that workaround can induce otherwise undesirable indirection that makes the code harder to understand.

Richard Feldman (Dec 08 2023 at 14:40):

I think the main pitch for adding this to the language would be:

  • it gives you an opt-in tool to make Roc code less error-prone in the rare circumstances where the tradeoffs around shadowing are the opposite from normal
  • it's a bit more verbose and less pretty, but that's actually good because it encourages using the simpler default (don't shadow) and only reaching for this when the alternative would be worse than the extra verbosity
  • unlike having shadowing enabled everywhere, beginners wouldn't have to learn about the "actually this isn't mutation, it doesn't work like mutable references do e.g. in the context of closures" etc. (Jose Valim mentioned that this is a thing some Elixir beginners find tricky) - it could be an advanced feature that's taught later
  • it's not introducing any new power to the language; you can already do all of this using other techniques; the problem is that they can make code harder to follow

yeah one way to think of the proposed design (with $ as the sigil) is:

  • Every time you name something with $ at the start of the name, the compiler essentially appends an incremented integer to the end. So the first time you define $foo it gets internally defined as $foo1, and the next time you define something named $foo it it gets internally defined as $foo2, etc.
  • whenever you reference one of these (e.g. myFunction $foo) it internally refers to the most recently defined one in scope (e.g. $foo1 or $foo2, whichever was defined more recently in scope)

Richard Feldman said:

and although it's true that reading and understanding code happens a lot more than writing it...

... being able to close off stale values and say "there's no chance the stale value will be referred to again after this point" is very valuable

Having fewer names to keep track of also helps with readability and comprehension. If the reader sees x1, x2, etc as a means to "name their way around shadowing restrictions," or in some ways worse, comes up with thoughtful names for each step, it can often give the reader the mistaken impression that the old names and new names need to be used side-by-side.

Some misc notes/anecdotes:

  • The Rust book specifically encourages shadowing via let instead of using mut whenever possible when the types are the same, and iirc doesn't discourage doing so when the type changes.
  • Most imperative language programmers are used to pass-by-value semantics for scalars alongside a pass-by-reference exception for certain types. Roc is strictly simpler due to the lack of that exception. In any case, except to day-1 new programmers, we won't need to teach that previously evaluated expressions do not get affected by subsequent assignments/shadowing.
  • In Python, it's been a historically pretty common practice to iteratively build up a value using a single variable, even when the type changes multiple times. For example, if you start with a string containing base 10 integers and want a sum, you might split it on spacing, assigning the list of strings over the same variable, and then map those into a list of integers, assigning over again, and then sum that list of integers, assigning over again. Python programmers would intuitively understand that all of those steps are part of the journey towards that final value, communicated specifically through reuse of the name. Python, at least at one point (in the 90s and early 2000s, when compared to other mainstream languages at the time), was known for encouraging practices that led to fairly few bugs despite having no static checking.
  • Your note about closing off stale values is analogous to closing off stale code flows in statement-oriented imperative languages through the use of if-return, if-continue, and if-break, rather than having an else which extends to cover the containing block. The conditionally truncated control has definitely been observed (at least by me) to improve code comprehension and reduce bugs, since at the point it happens, the reader has one less thing to keep track of, e.g. "this unusual case was handled with a return, so the rest of this function doesn't have to worry about it," instead of "well, I need to remember about this unusual case to see if might still apply after the end of this really long else, so I can either skip ahead to check now, or keep reading and hope I don't forget."
