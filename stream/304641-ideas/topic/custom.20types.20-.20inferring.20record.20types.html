Sam Mohr (Jan 02 2025 at 18:21):

For type inference of functions that might be custom type methods, it seems like we've got ambiguity.

module [U64Pair generating [to_str]]

U64Pair := { left: U64, right: U64 }

add_up : U64Pair -> U64
add_up = |pair| pair.left + pair.right

double_left = |item| item.left * 2

I'd assume the type of double_left is { left: Num a, .. } -> Num a. Since we don't have an active destructing syntax for custom values, it seems like we would need to say U64Pair has the following methods:

  • Any method generated by the compiler (to_str in this case)
  • Any function that has U64Pair as the type of the first arg
  • Any function whose first type is compatible with the inner type of U64Pair
    • Since { left: Num a, .. } is a supertype of { left: U64, right: U64 }, double_left is included

Sam Mohr (Jan 02 2025 at 18:23):

Or is that last case not allowed? In which case, we may want to consider one of the following changes:

  • prefixing methods with the type name, e.g. U64Pair.double_left = |item| item.left * 2
  • Having an explicit destructuring like double_left = |item.{ left, .. }| left * 2

Isaac Van Doren (Jan 07 2025 at 01:30):

I think if you want a function argument type to be a custom type, you should have to explicitly specify it in the function declaration.

# The type stays the same with or without the annotation
double_left : U64Pair -> U64
double_left = |U64Pair.item| # insert actual syntax here
    item.left * 2

If you omit the explicit identification of a custom type, then you get a structural record instead.

double_left : {left: U64}a -> U64 # plus the static dispatch part of the type for times
double_left = |item|
    item.left * 2

If you call a function that accepts a structural record with a custom type and the fields required by the function are exposed, then all is well. If they are not exposed, then it is a type error.

double_left : {left: U64}a -> U64
double_left = |item|
    item.left * 2

x = double_left(U64Pair.new { left: 1, right: 1}) # Ok if `U64Pair.left` is exposed, error if it is not

If you call a function that accepts a custom type with a structural record, that is a type error.

double_left : U64Pair -> U64
double_left = |U64Pair.item| # insert actual syntax here
    item.left * 2

x = double_left({left: 1}) # type error

This maintains the property that adding or removing a type annotation won't change if your code compiles and it doesn't require the addition of a notion of opaque types in addition to custom types. It feels natural to me also. Thoughts?

Richard Feldman (Jan 07 2025 at 03:00):

Sam Mohr said:

I think the original discussion was that we wanted to make a simple way to define concrete types that can be passed over FFI worry-free

yeah, the motivations are:

  • we no longer support recursive structural types, and opaque types shouldn't be allowed to cross the host boundary. Putting the two together, it is currently impossible to send recursive types to the host, which is a limitation I would like to remove.
  • There are a number of features custom tag unions in particular would unlock, such as Bool being able to be just True and False again (while still having custom serialization logic, which was not possible back when it was a structural type), Result being able to have methods run on it in the static dispatch world (e.g. my_result.map_err(...) working), being able to customize numbers/strings associated with different tags in a custom union (e.g. error codes), being able to mark tag unions as non-exhaustive so they can be added to without breaking backwards compatibility, etc.
  • Nominal tuples would allow for matrix types whose fields can be accessed using .0, .1, etc. while also being able to work with operator overloading and other customizations
  • Opaque wrappers work fine as a way to hide implementation details, but they seem to be harder to understand than e.g. Rust's public/private fields on structs
