Richard Feldman (Jan 08 2025 at 23:18):

one of Roc's nicest unique features is being able to decode based on return type, e.g. 

user = File.read(path, Json.utf8)?

I consider it absolutely essential that we preserve this capability in the static dispatch world

Richard Feldman (Jan 08 2025 at 23:19):

in the static dispatch world, something we haven't settled on is:

  • what does the type look like for functions where we're dispatching on return type instead of on the first argument? (first argument being the much more common "method style")
  • what does the syntax look like at the expression level for doing this? Like we have method syntax for saying "go look up this type's module and pass it as the first argument" but what if we want to go look up the return type's module? What's the expression-level syntax for that?

Richard Feldman (Jan 08 2025 at 23:24):

and if you don't have a type annotation, *.foo(bar) would basically say "I'm looking for a module that exposes a foo function which accepts bar as its only argument, and I'm going to find that module by looking up what module the return type of *.foo(bar) is defined in"

Richard Feldman (Jan 08 2025 at 23:26):

in this design, we'd have two ways to express static dispatch constraints:

    where a.decode(List U8) -> Something

...means "the module where a is defined exposes a function named decode which takes an a as its first argument and List U8 as its second argument, and returns a `Something"

    where *.decode : List U8 -> a

...means "the module where a is defined exposes a function named decode which takes a List U8 and returns an a"
