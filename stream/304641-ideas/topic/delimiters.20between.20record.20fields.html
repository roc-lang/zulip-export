Topic: delimiters between record fields

Richard Feldman (Mar 22 2023 at 15:51):

something I've noticed about records which take functions (e.g. event handlers) is that writing inline lambdas for them is annoying because it requires parentheses:

things = {
    blah: Foo,
    bar: "stuff",
    onWhatever:
        (
            \event, arg2 ->
                when event is
                    Whatever -> "whatever"
                    _ -> "other"
        ),
    onSomething:
        (
            \event, arg2 ->
                when event is
                    Something -> "something"
                    _ -> "other"
        )
}

Richard Feldman (Mar 22 2023 at 15:51):

what if you could write this instead?

things = {
    blah: Foo
    bar: "stuff"
    onWhatever:
        \event, arg2 ->
            when event is
                Whatever -> "whatever"
                _ -> "other"
    onSomething:
        \event, arg2 ->
            when event is
                Something -> "something"
                _ -> "other"
}

Richard Feldman (Mar 22 2023 at 15:52):

notice the lack of commas in between fields

Richard Feldman (Mar 22 2023 at 15:52):

the idea is that we use the fields themselves as the separator - e.g. when the parser sees bar: it knows that we must be done with the previous blah: definition

Richard Feldman (Mar 22 2023 at 15:53):

(currently it looks for a comma, but that's actually redundant; the comma must always be followed by either another field label or a })

Richard Feldman (Mar 22 2023 at 15:54):

I think we could also theoretically look for a trailing comma instead of requiring parens, like so:

things = {
    blah: Foo,
    bar: "stuff",
    onWhatever:
        \event, arg2 ->
            when event is
                Whatever -> "whatever"
                _ -> "other",
    onSomething:
        \event, arg2 ->
            when event is
                Something -> "something"
                _ -> "other"
}

Richard Feldman (Mar 22 2023 at 15:55):

however, I don't like this as much because:

  • it's very subtle and easy to miss (did you even notice the comma after the first _ -> "other",?) so if it's required, the errors could be confusing
  • it looks weird to me to see an expression that usually ends in a newline end in a comma instead - that is, _ -> "other",

Richard Feldman (Mar 22 2023 at 15:56):

a thing to consider with this idea: if we didn't have commas, would they become optional? That seems probably fine to me; the formatter could then remove them as a matter of style.

Richard Feldman (Mar 22 2023 at 15:57):

but then there's the question: what should the style be for single-line records? e.g. compare:

point = { x: 1, y: 2, z: 3 }

point = { x: 1 y: 2 z: 3 }

Richard Feldman (Mar 22 2023 at 15:57):

my initial reaction is that I prefer the first one, but honestly I bet I'd get used to the second one very quickly

Richard Feldman (Mar 22 2023 at 15:58):

so, to summarize - the idea would be:

  • make commas in between record fields optional, and use the fields themselves as separators
  • have the formatter remove the commas as a matter of style
  • this allows multiline expressions (including lambdas) in records to work without parentheses
