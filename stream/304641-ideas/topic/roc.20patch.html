Kiryl Dziamura (Jun 25 2025 at 09:03):

I want to propose a package patching/vendoring functionality built-in to the compiler.

what

package overloading functiobality that allows managing local changes of dependencies, including full vendoring

why

  1. a bug somewhere deep in dependencies. sometimes you need to just make it work not waiting for a new release (and a release of depended package)
  2. ability to modify internal packages for debugging

how

Introduce new commands

  • roc patch <package url> - fetches package contents to ./patches/<package>
  • roc patch collapse [package] - transforms the modifications to a diff file ./patches/<package>.diff

during compilation, roc either resolves imports from the local patched folder, or creates a new in a temp dir with applied diff and resolves from there

patches then possible to add to the index

uncertainties

  • nested package overloads: how to manage them?
  • should the patching be allowed only in applications?
  • patching introduces implicit indirection

alternatives

  • modify cached dependencies in the local roc registry: while it's good enough for debugging, it doesn't help with vendoring and also it invalidates cache for other applications

similar tools

https://pnpm.io/cli/patch

Richard Feldman (Jun 25 2025 at 13:55):

the way I think about it is:

  • today, we have no formal concept of versioning. everyone just gets the exact version of whatever dependency they say they want.
  • the only code sharing that happens is if multiple packages happen to specify that they want exactly the same version of a package, and it's obviously fine to code share in that circumstance because we verified via hash that they're all the same
  • in terms of correctness, this is all totally fine! There is a slight edge case where multiple packages expose functions that use opaque types from a dependency that they share, but since they're using different versions of that dependency, the opaque types are treated as incompatible types (since they were defined in different modules) - but from what I've seen in the Elm and Rust ecosystems, this happens so rarely in practice that I'm not really considering it a serious downside

Anthony Bullard (Jun 25 2025 at 13:55):

This is how Google's monorepo works.

Anthony Bullard (Jun 25 2025 at 13:55):

You have a third_party directory that contains _one, single version_ for every third party dep, and all internal code has a single version used by all other packages

Richard Feldman (Jun 25 2025 at 13:56):

however, there are downsides to only "exact match" code sharing like this:

  • potentially lots of unnecessary code duplication, which is a downside for both compile times and for final compiled binary size
  • if performance improvements, bug fixes, etc. are released, I don't have a way to get those updates to my indirect dependencies unless my direct dependencies release a new version which upgrades to them, and then I upgrade my direct dependencies
