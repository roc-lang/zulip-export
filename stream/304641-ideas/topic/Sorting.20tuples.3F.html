Aurélien Geron said:

Yes, it's commonly used in Python, and really quite handy, for example to compare versions (3, 1, 14) > (2, 9, 0), sort structured data (year, month, day), to sort events by timestamp, to use in data structures such as priority queues, and more. This lexicographical behavior is available in many other languages (Julia tuples, C++ with std::tuple, Rust tuples, etc.). I'm sure it's possible to live without it, but it would be good to have it IMHO.

quick design note - I want to draw a distinction between what I see as 3 different use cases:

Richard Feldman (Jun 19 2026 at 23:26):

  • sorting methods so data structures like trees can Just Work across things like tuples etc. I think this is critical - data structures that rely on sorting should Just Work with things like tuples, and for those it doesn't really matter what the semantics are in terms of humans; all that matters is that we can sort quickly and get consistent answers. I think we should definitely have and also auto-derive these methods, whatever names and semantics we end up choosing for them.
  • comparing things that are semantically their own types, like timestamps or versions. I get that in Python tuples are a reasonable choice for those, but I think in Roc they should definitely be nominal types! :smile: so I actually don't think it would be a selling point for us to facilitate doing it the Pythonic way instead of the Rockin' way :wink:
  • comparison operators like > and <=, which I think should be for numbers only. I think it's a footgun if string1 < string2 compiles. It's really easy to guess incorrectly what that might do, whereas number1 < number2 is extremely well-defined. I would put tuple1 < tuple2 in that category, and I think an example of this is that the first sentence of the first response to this thread is "How would that work?" :smile:

cc @Jonathan because you're working on #ideas > List sort builtins

Richard Feldman (Jun 19 2026 at 23:28):

so putting that together, I think:

  • tuples should absolutely be automatically sortable for purposes of going in data structures that rely on sorting their contents, like trees
  • I think the method name for this should be intentionally different from the method names the comparison operators desugar to, like is_gt and is_lt, so that < and >= intentionally don't work on tuples
  • I think timestamps and version numbers should be nominal types, and then of course if the authors of those nominal types want to add is_gt and is_lt methods on them, that's up to those authors :smile: (I do think it's a lot clearer what version1 <= version2 means, or timestamp1 <= timestamp2, compared to what those would mean with plain tuples on either side of the operator!)

Jonathan (Jun 24 2026 at 15:38):

So how about this as a summary/proposal:

  1. Order: Comparison functions should have a return type of [FirstBeforeSecond, Equivalent, SecondBeforeFirst], aliased here as Ord
  2. There is a convention for sorting and ordering, that when a default order is desired, it is given by the type module's default_cmp : a, a -> Ord method
  3. Primitives (as well as other types at their discretion) that do not have an obvious/unsurprising order, or are strongly coupled to presentation, should not have default_cmp defined
  4. Primitives that do have a clear ordering, or where one can be defined without reasonable alternative, are: Integers, tuples, Decimals(?), and (as a separate but related proposal) something like "Bytes" - a new type to denote/wrap strings where the notion of locality has been explicitly opted-out of.
  5. Sorting: If using the list sort api with a default comparison, the 'defaultness' of this is indicated by sort_with_default(List(a)) where [a.default_cmp : a, a -> Ord].
  6. If you want to sort by a codomain that does have a default_cmp, you use sort_by(List(a), mapper_fn) where [mapper_fn : a -> b, b.default_cmp : b, b -> Ord].
  7. If you want to supply your own comparison function, you use sort(List(a), comp_fn)
  8. Beyond sorting: It may seem like the "default comparison" is quite anemic now and could be removed, but default comparisons also serve to allow simpler data structure APIs
