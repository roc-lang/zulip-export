Richard Feldman (Oct 13 2023 at 14:43):

some relevant context about customizing infix operators:

  • Elm and Haskell used to let you define new infix operators (e.g. |+|), but not redefine builtin ones (such as +) but then Elm removed that capability after the way they were used seemed like a net negative. I think that was a good decision and I don't want to introduce arbitrary new operators to Roc.
  • Similarly, C++ allowed overloading of builtin operators (e.g. +) and James Gosling intentionally removed that capability for Java based on how he saw it used in C++ (e.g. repurposing the bit shift operator for things like cout << "Hello, World!")
  • Rust allows customizing what + and << do, but I haven't really seen it used in the same way it seems to be in C++. I'm not sure if that's a difference of cultural norms or something else.
  • The way numbers are defined today in Roc's type system, allowing user-defined types to work with + would require some sort of compiler change, and potentially a change to the design of the type system. So there's a significant bar to clear to justify adding that to the language.
