Anton (Jan 13 2025 at 13:16):

This is going to look like I'm shitting on static dispatch again, which I don't enjoy, because many people are excited about it and have put in substantial work towards it.

Given what we know now, I'd like us to reflect back on static dispatch. It seems like the SD decision has more downstream consequences and is more disruptive then we expected. There has been a ton of discussion about SD related things, which implies it's not easy to get the details right. I'm also concerned that as we make ever more changes to create a nicely balanced language with SD it will be harder to decide to go back, in case we think the final transformation is not better then the old Roc. A looming 0.1 release and the general desire to be respectful and appreciative of others work also decrease the likelihood of reverting.

So given this additional info, now seems like good time to make sure we still want to do this!
We had a simple well balanced language before SD and it looks like we're taking a substantial risk now.

To remove some doubt on this communication through text, I'm not angry and I don't feel strong emotions about the static dispatch situation. I'm just rationally trying to help us make the best language.

Brendan Hansknecht (Jan 13 2025 at 19:56):

As for pros and cons of SD...

Pros:

  • More autocomplete friendly (and mainstream syntax)
  • Replaces abilities with something simpler and easier to use/understand
  • Adds implicit interfaces (which enables very flexible composition and interoperability)
  • Removes a lot of repeated verbosity with Module names
  • Is still static calls (important for performance)
  • Tangential, but important: Comes with nominal types which enable a lot more flexibility in use than opaque types (like allow for pattern matching over nominal tags).

Cons:

  • Removes a significant part of the unique and joyful part of Roc's syntax that made it different from other languages (space based calling)
  • Could easily lead to too many nominal types (Though this is hard to quantify. In many languages there are only nominal types. Is having more really a problem)
  • More dependence on lsp "go to definition" to figure out an implementation
  • Awful default types for functions using static dispatch without concrete types (I really think forcing typing with an interface definition would be much more readable)
  • Could be totally avoid with many of the gains by just using records of functions and data. This would be dynamic dispatch and more verbose, but it would solve most of the same problems.

Richard Feldman (Jan 13 2025 at 21:24):

anyway, here's my list:

Pros:

  • Removing Abilities from the language
  • Removing optional record fields from the language (in favor of method-call builders, e.g. Json.utf8.transform(CamelCase) over Json.utf8_with({ transform: CamelCase }))
  • Removing module params for the language (in favor of passing around small stateful values that can have lots of functions on them without needing to carry around a ton of closures at runtime, and still be used concisely)
  • Removing tuple extensibility from the language (in favor of something like .get_0() for tuple field access)
  • Improved autocomplete experience
  • Conciseness (e.g. more functions fit on the same screen without scrolling)
  • More obvious module organization, e.g. a nominal UserId obviously gets its own UserId.roc so it can get static dispatch, whereas today deciding whether a small type like that gets its own module is a source of unimportant debate
  • Every module's documentation becomes "types and values" and that's it (like Elm, and like Roc was before Abilities) - I'm listing this separately from the "removing Abilities from the language" because there's a separate impact on documentation beyond the "no longer something anyone needs to learn or think about" aspect (like there also is in removing optional record fields and tuple extensibility, although neither of those impact docs significantly)
  • Parens-and-commas syntax (which is intended to facilitate static dispatch) is nicer when calling zero-argument functions, which are much more common now that we have purity inference (e.g. Dict.empty() compared to today's Dict.empty {})
  • Operator overloading without having to change anything's type (e.g. Num.plus can keep Num.add's current Num-focused type instead of having to become something like Num.add : a, a -> a where a implements Add)
  • It makes ad-hoc polymorphism structural; that is, two packages which share no dependencies can adhere to a common "protocol" involving static dispatch types of the same shape. Today, they can only have a common "protocol" by having a dependency on the same formal Ability. (If formality is desired, this can still be achieved by including an opaque type in the shared function shape; that way, anyone who wants to conform has to depend on that opaque type. The opaque type can even be a phantom type or zero-sized marker value.)
  • Static dispatch seems easier to teach than Abilities. (I already tried writing the tutorial section on the basic method-style calling version of it; it takes about a page of explanation, including walking through examples. The Decoding explanation will take longer—as it would with Abilities—but that can be deferred until later in the learning experience.)
  • 4.(hours) and 4.(hours).(ago!) are very nice imo, and one of the things people love about Ruby syntax (except in this case with full type-checking instead of monkeypatching)
  • Larger community and ecosystem, due to more people trying the language because syntax looks more mainstream (likely a lot more, based on Gleam's experience report). I think this is a very significant benefit, but I listed it last because I've seen occasional reactions to people hearing about static dispatch as a knee-jerk "that must be only for beginner familiarity!" so I put it at the end of a long list of unrelated upsides to make it more obivous that it's a benefit but not the benefit.

Cons:

  • More parens in the syntax (unavoidable downside that comes with the upside of more mainstream calling syntax)
  • Less flexibility in where ad-hoc polymorphism is defined (e.g. you can't have a nominal UserId in the same module as User and have them both support ==; unavoidable downside that comes with the upside of more obvious module organization)
  • Since ad-hoc polymorphism is based on function names, function names become more semantically important and it becomes possible to accidentally choose a name which later turns out to conflict with seemingly-unrelated code. (This apparently is not a significant problem in practice in Go, which has a similar consideration with their Interfaces. This is an unavoidable downside that comes with the upside of structural ad-hoc polymorphism, and is a problem which could also theoretically happen with our structural records, tuples, and tag unions, but so far has not.)
  • Less self-descriptive code because of fewer fully-qualified calls (inlay hints can help with this in vertical call chains, but still not the same as before; unavoidable downside that comes with the upside of conciseness)
  • We haven't found a great syntax for the types of e.g. Decode (static dispatch in the return position instead of the argument position)
  • It's potentially easier for beginners to continue using familiar object-oriented architectures that don't fit the language (unavoidable downside that comes with the upside of code looking more familiar to beginners)

Paul Stanley (Jan 13 2025 at 22:59):

I don't particularly feel qualified to have an opinion, and I'm always in favour of trying things out. That said for me the main cons are that static dispatch brings with it:

  • A good deal of obscure magic. Yes, fully qualified functions are annoying sometimes, but they do really increase expressness in a good way. There's a reason why it's a generally held wisdom in, e.g., OCaml _not_ to open modules globally. I think people are going to find some odd surprises, and it's going to be very hard to make the LSP story so good that they disappear.
  • I dislike treating functions as if they were methods. To me foo.dobar(blah).doobaz(blahblah) looks as if dobar and dobaz are in someway features of foo. But they are not, and actually they (or, anyway, any call other than the first) may be operating on things that have a completely different type from foo. Pipelines seem a better metaphor to me (and even they can be over-used). map is not a thing a list can do it is a thing you can do to a list, and it's first argument is no more semantically significant than its second.
  • I happen not to like syntax where a simple little dot means "call a function". The . (not a very conspicuous little fellow) ends up massively overloaded: sometimes it means decimal point, sometimes it means record field access, sometimes it means call a function.
  • I've never loved a proliferation of parentheses. One of the reasons ML type languages end up with pipe operators is to avoid them. I don't hate them is Lisp, but there they come alongside an elegantly simple s-expression syntax. In the end, they too end up "overloaded" with meaning, sometimes signalling grouping, sometimes signalling function arguments ... it's a mess. The fact it's a familiar mess in lots of languages makes us not notice the mess, but it's a mess.

So personally those are downsides which for me outweigh the upsides. But I say that as a total amateur _and_ someone who has spent much more time with functional languages, especially ML-lie ones, so _for me_ the existing patterns are more familiar.

Disagree and commit is a method of avoiding the consensus trap, in which the lack of consensus leads to inaction.
