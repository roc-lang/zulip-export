Topic: ✔ Static Dispatch - without nominal typing?

Anthony Bullard (Dec 28 2024 at 13:52):

I was reading through the static dispatch proposal for the 6th time, and I started having the thought - a thought I had a couple of years ago developing my own language - couldn't we have static dispatch that considers any _in scope_ top-level function that takes this type as it's first argument?

  • LSPs could do the work of finding/importing/exposing the correct member in autocomplete
  • All functions that are eligible would have to be exposed unqualified in the module
  • The need for a focus on nominal typing, compiler magic, and loss-of-understanding goes away, but the same developer experience
  • Static dispatch already relied on LSP/tooling for it's benefits to DX.
  • This system would have arguably the same or better experience for those that don't use LSP.
  • The nominal/custom types would have all the same behavior as with Richard's original proposal, but it would not favor nominal over the structural types that make Roc feel special.

See #ideas > Static Dispatch - without nominal typing? @ 💬 below to see an example of the workflow.

The biggest disadvantage is that it may mean that abilities may have to stick around / have changes
