Topic: harmonize decision structures

Matthias Toepp (Apr 24 2023 at 05:48):

IDEA:

  1. Harmonize if with when..is syntax.
  2. Use else at the end of all decision structures.

BENEFITS:

  1. Enhance the feeling of consistency when using roc's decision structures:

    • adds a kind of zen quality to roc.
    • using else as the last branch of all decision structures will feel more consistent
    • the currentif..else and when..is would change to look/feel/handle in much more the same way.

  2. Make roc feel more conventional AND intuitive AND robust, by having else at the end of decision structures, including at the end of when..is:

    • "_->" can feel like strange syntax, beginners would not need to process that strangeness. The meaning of else is both more conventional and more intuitive than _->.
    • else is more visible and feels more robust than "_->".

  3. Make roc cleaner in terms of noise & alignment (with the proposed multi-branch Boolean structure)

    • conditions would get the focus of attention that they deserve by being placed front and center, rather than imbedded in noisy syntax (after else if and before then).
    • vertical scanning of conditions would become more natural for the eye as the conditions would be vertically aligned. Currently the conditions are pushed five characters out of vertical alignment with each other in an else..if structure.

  4. Enhanced refactorability:

    • The first lines in the proposed multi-conditional structure could be more easily reordered.
    • it would be slightly easier to add a new condition above the first condition.

  5. Enhanced diff.

    • ordering does not rely on a change of if to else if - so changes there would probably be more accurate in regards to version control diffs. (quoting @Georges Boris )

  6. This syntax may help to ease people into learning pattern matching, as the parallels to a multi-branch boolean structure are more apparent when the syntaxes are more similarly laid out.

THE PROPOSED SYNTAX:

# `when..is`: multi-line branching on pattern matching:
size =
    when x is
        Small ->
            "small"
        Medium ->
            "medium"
        else              # note use of else is similar to next example.
            "large"

# `when`(used alone): multi-line branching on boolean expressions:
size =
    when
        x < 10 then
            "small"
        x < 100 then
            "medium"
        else               # note use of else is similar to prev. example.
            "large"

# single line use:
# refactors nicely to the above.
size =
    when x < 10 then "small" else "large"

DISADVANTAGES:

  1. Requires an extra line vertically for the multiline syntax.

  2. More unusual to people who have already learned another language with if..else syntax.

    • Arguably though, seeing the if..else expressed this way moves you part-way towards understanding pattern matching syntax in terms of something that you already understand.

  3. Requires an extra level of indentation for multiline syntax.

    • It is debatable if this is a problem at all or just a problem anyway regardless of this syntax...roc's indenting convention may or may not need to changed anyway with or without this syntax.
      https://roc.zulipchat.com/#narrow/stream/304641-ideas/topic/less.20indentation.3F

    val num = 5
    val message = if (num % 2 == 0) "even" else "odd"
    println("$num is $message") // => 5 is odd

    // "when" can be used as an alternative to "if-else if" chains.
    val i = 10
    when {
        i < 7 -> println("first block")
        fooString.startsWith("hello") -> println("second block")
        else -> println("else block")
    }

    // "when" can be used with an argument.
    when (i) {
        0, 21 -> println("0 or 21")
        in 1..20 -> println("in the range 1 to 20")
        else -> println("none of the above")
    }

    // "when" can be used as a function that returns a value.
    var result = when (i) {
        0, 21 -> "0 or 21"
        in 1..20 -> "in the range 1 to 20"
        else -> "none of the above"
    }
    println(result)
