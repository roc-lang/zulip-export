One pattern that comes up here and there in our Elm code base at work is: for any given custom type, where none of the constructors have payloads, get a list with each constructor in. I.e.:

MyTags : [Foo, Bar]

list : List MyTags
list = [Foo, Bar]

The main use we have for this pattern is for showing simple selections in our UI. That is, if we have a dropdown with a fixed set of options, it would be good to represent those as actual types instead of just strings for example.

To me, this problem is kind of similar to that of Eq, where the compiler provides default implementations in some cases today. It does so because maintaining your own implementation is error prone, as it can easily get out of sync, and it's easy to build it automatically.

So maybe there could be e.g. a Listable ability which the compiler autogenerates for applicable types? That strikes a nice balance to me, though as with Eq, it does have the problem that a library author could implicitly make a breaking change if they change a type such that the compiler no longer generates these implementations without the author realizing it.
