Topic: list of all tags in a union

Richard Feldman (Jan 30 2022 at 22:49):

a thing that I've seen come up in languages with enumerations is wanting to get a list of all the different values in the enumeration so you can, for example, make a dropdown out of them. So in Roc it might be like Color : [ Red, Green, Yellow, Blue, Purple ] and then I want to get a List Color which has one element for each of the different tags in the tag union

Richard Feldman (Jan 30 2022 at 22:49):

I think it's worth Roc having a language feature like this; the question is how it should look/work

Richard Feldman (Jan 30 2022 at 22:50):

here's one idea:

Color : [ Red, Green, Yellow, Blue, Purple ]

allColors : List Color
allColors = ...Color

so, ... in front of a type means "if this is a closed tag union with no payloads, give me a List consisting of one of each of its elements"

Richard Feldman (Jan 30 2022 at 22:52):

it's pretty specific syntax, but I haven't seen a nice alternative that keeps the type and the list in sync - there's a technique you can use to get a compiler error if add a new element to the list and forget to remove an old one, but it's be easy to make a copy/paste error when constructing or modifying the when, which would mean they'd be out of sync and the compiler wouldn't be able to help

Richard Feldman (Jan 30 2022 at 22:52):

the reasoning behind ... in particular is wanting to use something that won't interfere with autocomplete

Martin Stewart (Jan 31 2022 at 08:40):

Richard Feldman said:

I think I'd probably say sort them alphabetically

I think sorting by definition order would be less error prone. If I do a rename I’m probably not thinking about how that might reorder a list elsewhere. Alternatively what if there was no syntax for enumerating all tags, but there was a containsAllTags function? You could use that with expect to catch yourself when you forget to add a new tag. This has the advantage of not needing new syntax and makes sure the user is explicit with their tag ordering.
