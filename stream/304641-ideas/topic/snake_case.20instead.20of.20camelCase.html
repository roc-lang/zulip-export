Sky Rose (Nov 10 2024 at 14:37):

Assuming we have the formatter convert from one to the other (in either direction), one gotcha we'll need to avoid: If you define both fooBar and foo_bar, then the formatter needs to not convert, because that would be a logic change. You'd need a human to rename one to resolve the conflict.

But if you define fooBar and then reference foo_bar, should the compiler tolerate that, and should the formatter rename one to match the other? It'd be similar to the Nim problem, except with the assumption that when you eventually run the formatter there will be one correct version. It'd also be a very common case, for example if you're trying to call a standard library function with the wrong casing.

Here's something that doesn't have to do with the actual format itself. Having the formatter fix the casing of identifiers means that this is somewhere where the AST will be changed by the formatter, which is currently not allowed. I can think of two ways of handling this:

  • Change the AST changed determination to allow these casing changes (would require some pretty wonky changes)
  • Pass something like --migrate flag to the format command that allows for the formatter to change the AST. Elixir did this recently in 1.17 for exactly the same reason(allow for changes to AST in formatter, not casing as that has also been the case)l

I'm leaning towards the latter as it is simpler. We could also probably add a confirmation prompt for this. Richard Feldman (Nov 30 2024 at 15:07):

my personal view on this is:

  • we should continue to use capitalization as a concise way to distinguish between things, e.g. types vs type variables, records vs modules, tags vs variables
  • we should continue to use PascalCase for types, tags, and module names, because Pascal_Snake_Case is not a thing any language uses and it's not worth the strangeness budget cost
  • given that, a selling point of camelCase over snake_case is that its delta with PascalCase is smaller (it only differs in the capitalization of the first letter)
  • it's also fine to pair snake_case with PascalCase in the same language - e.g. Ruby and Rust both do it - and although consistency is an objective benefit of camelCase, subjectively there still seems to be a strong preference for snake_case.
