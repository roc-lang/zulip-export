Richard Feldman (Sep 20 2024 at 16:10):

the reason var is important is for the following scenario:

  • let's suppose there is no var, and the underscore suffix means that it's reassignable
  • I have some existing code written that uses foo_ and reassigns it, as normal.
  • In a nested conditional where foo_ is in scope, I start writing some new code. I happen to also use the name foo_ for this new code, not realizing that I am referring to a foo_ that already exists in the current scope.
  • Bugs! I am now changing how the code in the outer scope works even though I didn't intend to.
  • There is no possible compiler error to help me out, because I'm doing something that's totally allowed.

var fixes that because now if I'm introducing (what I believe is) a new usage of foo_ in the nested scope, I'll start by writing var foo_ and immediately get a shadowing error
