Topic: Maybe tag/record extension could be more expressive?

Austin Davis (Jul 23 2025 at 20:13):

Right now we have some flexibility w/ regard to tag/record extensibility. We use the following syntax

TagA x : [ A ]x
TagAB : [ B ]TagA

RecordA x : { a : Str }x
RecordAB : { b: : Str }RecordA

On the plus side, this really does help with extensibility and polymorphism when you need it. But one downside of how this is implemented is that (unless I'm missing something), you cannot easily extend multiple algebraic types in the same type definition.

For example, I may want to write something like this:

TagA : [ A ]
TagB : [ B ]
TagC : [ C ]

TagABC : [ ...TagA, ...TagB, ...TagC ]
# TagABC : [ A, B, C ]

Technically we can do something like this:

Tag A x : [ A ]x
Tag B x : [ B ]x
Tag C x : [ C ]x

TagABC : TagC (TagB (TagA []))

But that can get unwieldy very quickly, especially since each of the smaller tag unions is required to have its own type variable. And on top of that, this kind of composition doesn't currently allow for recursive structures, although perhaps that's technically possible and just not implemented yet.

My main use case right now is to model an AST for a language parser. I want to be able to have something like this:

ArrayExpressionNode : [
    ArrayExpression { loc : Location, elements : List ExpressionNode }
]

AssignmentExpressionNode : [
    AssignmentExpression { loc : Location, operator : AssignmentOperator, left : [...PatternNode, ...ExpressionNode], right : ExpressionNode },
]

LogicalExpressionNode : [
    LogicalExpression { loc: Location, operator : LogicalOperator, left : ExpressionNode, right : ExpressionNode },
]

ExpressionNode : [
    ...ArrayExpressionNode,
    ...AssignmentExpressionNode,
    ...LogicalExpressionNode,
]

As it stands, I can almost accomplish this using the following tag-extension syntax:

ArrayExpressionNode a : [
    ArrayExpression { loc : Location, elements : List ExpressionNode }
]a

AssignmentExpressionNode a : [
    AssignmentExpression { loc : Location, operator : AssignmentOperator, left : [Pattern PatternNode, Expression ExpressionNode], right : ExpressionNode },
]a

LogicalExpressionNode a : [
    LogicalExpression { loc : Location, operator : LogicalOperator, left : ExpressionNode, right : ExpressionNode },
]a

ExpressionNode : ArrayExpressionNode (AssignmentExpressionNode (LogicalExpressionNode []))

But the recursive reference to ExpressionNode within the other three tag union definitions seems to cause the compiler to crash. I know there are some seemingly intentional restrictions on type-level recursion in Roc, and perhaps those restrictions will always remain in place. But I cannot seem to find a way to produce the recursive type structure that I need. For example, this fails with a stack overflow at compile time:

ArrayExpressionNode : [
  ArrayExpression { loc : Location, elements : List ExpressionNode }
]

AssignmentExpressionNode : [
    AssignmentExpression { loc : Location, operator : AssignmentOperator, left : [Pattern PatternNode, Expression ExpressionNode], right : ExpressionNode },
]

LogicalExpressionNode : [
    LogicalExpression { loc : Location, operator : LogicalOperator, left : ExpressionNode, right : ExpressionNode },
]

ExpressionNode : [
    ArrayExpression ArrayExpressionNode,
    AssignmentExpression AssignmentExpressionNode,
    LogicalExpression LogicalExpressionNode,
]

In the above example, it seems that the only recursive reference to ExpressionNode that doesn't cause the compiler to crash is the List ExpressionNode type located within ArrayExpressionNode. Tag wrappers like [Pattern PatternNode, Expression ExpressionNode] actually cause compiler crashes. I assume this isn't intentional. And the recursion issue probably deserves its own topic.

But regardless, even if the above code worked, it is still somewhat annoying to have to match on a pattern like this:

get_loc : ExpressionNode -> Location
get_loc = |expression|
    when expression is
        ArrayExpression(ArrayExpression({ loc })) -> loc
        AssignmentExpression(AssignmentExpression({ loc })) -> loc
        LogicalExpression(LogicalExpression({ loc })) -> loc

And that excessive boilerplate is what I hope Roc can help me avoid eventually...

Also, I should say that I can currently work around this problem by simply duplicating a lot of type signatures (which I think is probably unnecessary from a technical standpoint):

ArrayExpressionNode : [
    ArrayExpression { loc : Location, elements : List ExpressionNode }
]

AssignmentExpressionNode : [
    AssignmentExpression {loc: Location, operator : AssignmentOperator, left : [Pattern PatternNode, Expression ExpressionNode], right : ExpressionNode },
]

LogicalExpressionNode : [
    LogicalExpression { loc : Location, operator : LogicalOperator, left : ExpressionNode, right : ExpressionNode },
]

ExpressionNode : [
    ArrayExpression { loc : Location, elements : List ExpressionNode },
    AssignmentExpression {loc: Location, operator : AssignmentOperator, left : [Pattern PatternNode, Expression ExpressionNode], right : ExpressionNode },
    LogicalExpression { loc : Location, operator : LogicalOperator, left : ExpressionNode, right : ExpressionNode },
]

get_loc : ExpressionNode -> Location
get_loc = |expression|
    when expression is
        ArrayExpression({ loc, elements }) -> loc
        AssignmentExpression({ loc, operator, left, right }) -> loc
        LogicalExpression({ loc, operator, left, right }) -> loc

This gives me a way to have aliases for individual tags as well as an alias for larger tag unions that contain those individual tags. This avoids type-level mutual recursion, so it doesn't cause a compiler crash, and it's a relatively easy API to consume.

But as you can see, it requires manually ensuring that all the duplicated types remain identical to each other, which is not ideal from a maintenance standpoint.

Brendan Hansknecht (Jul 23 2025 at 20:53):

Two fly by comments as I haven't had time to dive into this fully:

  1. We plan to switch to the [A, B, ...SomeTag] syntax for expanding tags
  2. In the future all recursive data structures will require being through nominal types. Which require being explicit and do not allow expansion. With the only exception maybe being List based recursion (and maybe box, but that requires a null case, so it not as clear if it requires being nominal or not).
