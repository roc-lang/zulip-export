Topic: Default Valued Record Fields in Type Annotation

Brendan Hansknecht (Feb 25 2024 at 18:16):

So @Brian Teague brought this up, and I think it is likely how we should handle Default Valued Record Fields.

Simply put, a user can add a default value to any field of a struct in the type definition. Ignore the exact syntax, but something like:

Point : { x: F32, y: F32, z: F32 = 0.0 }

The reads as: We have a type Point. It contains the floats: x, y, and z. z has a default value of 0.0.

I think this fixes the inconsistencies and confusion around optional record fields while also being a more generally useful concept.

For starters, throw a roc at any systems level programming language (and many higher level ones), and they have this feature. Default initialization is generally useful. It becomes exceptionally powerful with configuration structs where users will mostly use the default initialization of fields, but might override the initialization of a few specific fields.

Rust:

Foo {
    bar: "bar".to_string(),
    baz: 42,
    ..Default::default()
}

Zig:

const Foo = struct {
    a: i32 = 1234,
    b: i32,
};

C/C++:

struct foo {
    bool a = true;
    bool b = true;
    bool c;
 } bar;

I think that this:

  1. Changes the name to default record field instead of optional record field which will clear the common beginner confusion.
  2. It no longer requires destructuring the record.
  3. It is more flexible and powerful.
  4. Still only gives the power of default values with names.

Downsides:

  1. Requires a type annotation to use. That said, it will never break our type decidablility. People will just have to explicitly fill out all types if an annotation doesn't exist.

Here a list of how other language are doing this (generated by Chat GPT):
Sure, here's the updated list including Kotlin and Scala:

  1. Kotlin: Kotlin supports default parameters in a straightforward manner. If a value for a parameter is not provided, the default value is used.
fun greet(name: String = "World") {
    println("Hello, $name!")
}

greet()  // prints "Hello, World!"
greet("Alice")  // prints "Hello, Alice!"
  1. Scala: Scala also supports default parameters. If a value for a parameter is not provided, the default value is used.
def greet(name: String = "World"): Unit = {
    println(s"Hello, $name!")
}

greet()  // prints "Hello, World!"
greet("Alice")  // prints "Hello, Alice!"
  1. Python: Python supports default parameters in a straightforward manner. If a value for a parameter is not provided, the default value is used.
def greet(name="World"):
    print(f"Hello, {name}!")

greet()  // prints "Hello, World!"
greet("Alice")  // prints "Hello, Alice!"
  1. JavaScript (ES6 and later): JavaScript supports default parameters using the assignment (=) operator in the function definition.
function greet(name = "World") {
    console.log(`Hello, ${name}!`);
}

greet();  // prints "Hello, World!"
greet("Alice");  // prints "Hello, Alice!"
  1. PHP: PHP also supports default parameters. If a value for a parameter is not provided, the default value is used.
function greet($name = "World") {
    echo "Hello, $name!";
}

greet();  // prints "Hello, World!"
greet("Alice");  // prints "Hello, Alice!"
  1. Swift: Swift supports default parameters as well. If a value for a parameter is not provided, the default value is used.
func greet(name: String = "World") {
    print("Hello, \(name)!")
}

greet()  // prints "Hello, World!"
greet(name: "Alice")  // prints "Hello, Alice!"
  1. C#: C# supports optional parameters that can be used as default parameters.
public void Greet(string name = "World") {
    Console.WriteLine($"Hello, {name}!");
}

Greet();  // prints "Hello, World!"
Greet("Alice");  // prints "Hello, Alice!"
  1. Rust: In Rust, you can define functions with default parameters by using the Option type. Here's how you can define a function with default parameters in Rust:
fn greet(name: Option<&str>, age: Option<i32>) {
    let name = name.unwrap_or("World");
    let age = age.unwrap_or(20);
    println!("Hello, {}! You are {} years old.", name, age);
}

In this case, if you call greet(None, None), it will print "Hello, World! You are 20 years old.". If you provide an argument, like greet(Some("Alice"), Some(30)), it will print "Hello, Alice! You are 30 years old.".

However, Rust does not support named arguments like Kotlin and Scala. You have to pass the arguments in the order they are defined in the function signature.

These are just a few examples. Many other languages also support this feature.

Kevin Gillette (Mar 08 2024 at 14:07):

What I am worried about is mixing value notions into a type definition, because values and types are distinct in Roc. If we're going to bend that, then why not just implement full dependent type support? It's a slippery slope.

Additionally, I'm worried about the data modeling abuse that would inevitably come from being able to specify defaults within a named type. We shouldn't have a Point with an optional z. We should have Point2D and Point3D instead.

Kevin Gillette (Mar 08 2024 at 14:24):

iow, I'd feel better if Roc disallowed optional fields to be used in named types, limiting their use only to unnamed types used in specific contexts (like input to a function call).

I'd feel even better if default/optional values were a value level concept rather than also being a type-level concept, if that could be done while still having the default value feature without harming or special-casing this aspect of the type system.

iow, something like:

add : { x : Num a, y : Num a } -> Num a
add = \{ x, y ? 1 } -> x + y

five = add { x: 4 }

I'm guessing, but I suspect that with today's syntax and semantics, the Roc compiler sees a call site like is used to define five above, and supplies an { x : Num * } to the function, thus requiring the type system to know about optional fields.

As mentioned above, I believe optional fields are bad for type modeling, and so my proposal would be that the compiler instead have the call site take default values from the function definition, thus desugaring the last line in the above snippet to:

five = add { x: 4, y: 1}

Thus we can have present-day optional/default field behavior without type system support.
